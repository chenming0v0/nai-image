## How to Use This Skill

When user requests UI/UX work (design, build, create, implement, review, fix, improve), follow this workflow:

### Step 1: Analyze User Requirements

Extract key information from user request:
- **Product type**: SaaS, e-commerce, portfolio, dashboard, landing page, etc.
- **Style keywords**: minimal, playful, professional, elegant, dark mode, etc.
- **Industry**: healthcare, fintech, gaming, education, etc.
- **Stack**: React, Vue, Next.js, or default to `html-tailwind`

### Step 2: Generate Design System (REQUIRED)

**Always start with `--design-system`** to get comprehensive recommendations with reasoning:

```bash
python3 skills/ui-ux-pro-max/scripts/search.py "<product_type> <industry> <keywords>" --design-system [-p "Project Name"]
```

This command:
1. Searches 5 domains in parallel (product, style, color, landing, typography)
2. Applies reasoning rules from `ui-reasoning.csv` to select best matches
3. Returns complete design system: pattern, style, colors, typography, effects
4. Includes anti-patterns to avoid

**Example:**
```bash
python3 skills/ui-ux-pro-max/scripts/search.py "beauty spa wellness service" --design-system -p "Serenity Spa"
```

### Step 2b: Persist Design System (Master + Overrides Pattern)

To save the design system for hierarchical retrieval across sessions, add `--persist`:

```bash
python3 skills/ui-ux-pro-max/scripts/search.py "<query>" --design-system --persist -p "Project Name"
```

This creates:
- `design-system/MASTER.md` — Global Source of Truth with all design rules
- `design-system/pages/` — Folder for page-specific overrides

**With page-specific override:**
```bash
python3 skills/ui-ux-pro-max/scripts/search.py "<query>" --design-system --persist -p "Project Name" --page "dashboard"
```

This also creates:
- `design-system/pages/dashboard.md` — Page-specific deviations from Master

**How hierarchical retrieval works:**
1. When building a specific page (e.g., "Checkout"), first check `design-system/pages/checkout.md`
2. If the page file exists, its rules **override** the Master file
3. If not, use `design-system/MASTER.md` exclusively

### Step 3: Supplement with Detailed Searches (as needed)

After getting the design system, use domain searches to get additional details:

```bash
python3 skills/ui-ux-pro-max/scripts/search.py "<keyword>" --domain <domain> [-n <max_results>]
```

**When to use detailed searches:**

| Need | Domain | Example |
|------|--------|---------|
| More style options | `style` | `--domain style "glassmorphism dark"` |
| Chart recommendations | `chart` | `--domain chart "real-time dashboard"` |
| UX best practices | `ux` | `--domain ux "animation accessibility"` |
| Alternative fonts | `typography` | `--domain typography "elegant luxury"` |
| Landing structure | `landing` | `--domain landing "hero social-proof"` |

### Step 4: Stack Guidelines (Default: html-tailwind)

Get implementation-specific best practices. If user doesn't specify a stack, **default to `html-tailwind`**.

```bash
python3 skills/ui-ux-pro-max/scripts/search.py "<keyword>" --stack html-tailwind
```

Available stacks: `html-tailwind`, `react`, `nextjs`, `vue`, `svelte`, `swiftui`, `react-native`, `flutter`, `shadcn`, `jetpack-compose`

## Common Rules for Professional UI

These are frequently overlooked issues that make UI look unprofessional:

### Icons & Visual Elements

| Rule | Do | Don't |
|------|----|----- |
| **No emoji icons** | Use SVG icons (Heroicons, Lucide, Simple Icons) | Use emojis like 🎨 🚀 ⚙️ as UI icons |
| **Stable hover states** | Use color/opacity transitions on hover | Use scale transforms that shift layout |
| **Correct brand logos** | Research official SVG from Simple Icons | Guess or use incorrect logo paths |
| **Consistent icon sizing** | Use fixed viewBox (24x24) with w-6 h-6 | Mix different icon sizes randomly |

### Interaction & Cursor

| Rule | Do | Don't |
|------|----|----- |
| **Cursor pointer** | Add `cursor-pointer` to all clickable/hoverable cards | Leave default cursor on interactive elements |
| **Hover feedback** | Provide visual feedback (color, shadow, border) | No indication element is interactive |
| **Smooth transitions** | Use `transition-colors duration-200` | Instant state changes or too slow (>500ms) |

### Light/Dark Mode Contrast

| Rule | Do | Don't |
|------|----|----- |
| **Glass card light mode** | Use `bg-white/80` or higher opacity | Use `bg-white/10` (too transparent) |
| **Text contrast light** | Use `#0F172A` (slate-900) for text | Use `#94A3B8` (slate-400) for body text |
| **Muted text light** | Use `#475569` (slate-600) minimum | Use gray-400 or lighter |
| **Border visibility** | Use `border-gray-200` in light mode | Use `border-white/10` (invisible) |

### Layout & Spacing

| Rule | Do | Don't |
|------|----|----- |
| **Floating navbar** | Add `top-4 left-4 right-4` spacing | Stick navbar to `top-0 left-0 right-0` |
| **Content padding** | Account for fixed navbar height | Let content hide behind fixed elements |
| **Consistent max-width** | Use same `max-w-6xl` or `max-w-7xl` | Mix different container widths |

- [ ] Test both modes before delivery

- [ ] Floating elements have proper spacing from edges
- [ ] No content hidden behind fixed navbars
- [ ] Responsive at 375px, 768px, 1024px, 1440px
- [ ] No horizontal scroll on mobile

- [ ] All images have alt text
- [ ] Form inputs have labels
- [ ] Color is not the only indicator
- [ ] `prefers-reduced-motion` respected

# NewAPI 绘图接口调用说明

本文档说明用户如何使用服务提供方给出的 NewAPI 兼容 OpenAI 地址和密钥调用绘图服务。

用户拿到的信息通常只有两个：

```text
OPENAI_BASE_URL = 服务提供方给你的 NewAPI 地址
OPENAI_API_KEY  = 服务提供方给你的 NewAPI 密钥
```

后续所有请求都发给 `OPENAI_BASE_URL`。

## 1. 请求头

所有请求都使用 OpenAI 兼容鉴权方式：

```http
Authorization: Bearer OPENAI_API_KEY
Content-Type: application/json
```

如果密钥错误、缺失或失效，通常会返回 `401`。

## 2. 获取模型列表

请求：

```http
GET OPENAI_BASE_URL/v1/models
```

curl 示例：

```bash
curl "OPENAI_BASE_URL/v1/models" \
  -H "Authorization: Bearer OPENAI_API_KEY"
```

响应示例：

```json
{
  "object": "list",
  "data": [
    {
      "id": "nai-diffusion-4-5-full",
      "object": "model",
      "created": 1700000000,
      "owned_by": "novelai"
    }
  ]
}
```

使用 `data[].id` 作为绘图请求中的 `model`。

常用模型：

```text
nai-diffusion-4-5-full
nai-diffusion-4-5-curated
nai-diffusion-4-full
nai-diffusion-4-curated
nai-diffusion-3
nai-diffusion-furry-3
```

## 3. 绘图接口

绘图请求发送到：

```http
POST OPENAI_BASE_URL/v1/chat/completions
```

注意：虽然接口路径是 OpenAI 的 `chat/completions`，但这里用于绘图。

不要调用：

```http
POST OPENAI_BASE_URL/v1/images/generations
```

当前绘图渠道不使用 OpenAI 图片接口格式，会直接返回 `501`。

## 4. 请求体规则

请求体外层使用 OpenAI Chat Completions 格式。

真正的绘图参数必须作为一个**合法的 JSON 对象字符串**放在：

```text
messages[0].content
```

也就是说：

- 外层 `model` 是 OpenAI 兼容字段，**这是模型名的唯一来源**。
- `messages[0].content` 必须是字符串，且 `JSON.parse` 后是一个 object。
- 这个 object 内部才是真正的绘图参数。

### 4.1 模型名规则（重要）

模型名以**外层 `body.model` 为准**。如果你在 `messages[0].content` 内部 JSON 里也写了 `model` 字段，会被服务端**静默忽略并覆盖**为外层值。

为了避免歧义，推荐内层 JSON 不再写 `model`。

### 4.2 content 格式（严格 JSON）

`messages[0].content` 必须是合法 JSON object 字符串。**纯文本会被直接拒绝**：

```jsonc
{ "role": "user", "content": "画一个女孩" }            // ❌ 400 INVALID_PAYLOAD
{ "role": "user", "content": "{\"prompt\":\"1girl\"}" } // ✅
```

也支持 OpenAI 多模态 content 数组，但**必须恰好包含一个 `type: "text"` part**，且不允许
出现 `image_url` 等其他类型的 part（出现即 400）：

```json
{
  "role": "user",
  "content": [
    { "type": "text", "text": "{\"prompt\":\"1girl\"}" }
  ]
}
```

### 4.3 最小请求示例

```json
{
  "model": "nai-diffusion-4-5-full",
  "messages": [
    {
      "role": "user",
      "content": "{\"prompt\":\"1girl, solo, masterpiece, best quality\",\"negative_prompt\":\"lowres, bad anatomy\",\"size\":[832,1216],\"steps\":23}"
    }
  ],
  "stream": false,
  "max_tokens": 100000
}
```

代码里**不要手写转义字符串**，应该用语言自带的 JSON 序列化（Python 的 `json.dumps`）：

```python
import json

draw_params = {
    "prompt": "1girl, solo, masterpiece, best quality",
    "negative_prompt": "lowres, bad anatomy",
    "size": [832, 1216],
    "steps": 23,
}

request_body = {
    "model": "nai-diffusion-4-5-full",
    "messages": [
        {"role": "user", "content": json.dumps(draw_params)},
    ],
    "stream": False,
    "max_tokens": 100000,
}
```

## 5. 绘图参数

`messages[0].content` 内部 JSON 支持以下字段（不需要的字段全部省略即可）：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `prompt` | string | 是 | 正向提示词，必须使用英文。 |
| `negative_prompt` | string | 否 | 负向提示词，必须使用英文。 |
| `size` | `[width, height]` | 否 | 整数数组。**不支持** `"832x1216"` 这种字符串写法，也**不支持** `"portrait"` 等预设名。缺省 `[832, 1216]`。 |
| `steps` | number | 否 | 迭代步数，最大 `28`，缺省 `23`。 |
| `scale` | number | 否 | 提示词引导强度，常用 `5.0`。 |
| `sampler` | string | 否 | 采样器，缺省 `k_euler_ancestral`。 |
| `seed` | number | 否 | 随机种子；省略 / `null` 表示随机。 |
| `n_samples` | number | 否 | 生成张数。**当前只允许 `1`**，传 `>1` 直接 400。建议直接省略。 |
| `variety_boost` | boolean | 否 | 对应官网 Variety+ 开关。 |
| `cfg_rescale` | number | 否 | Prompt Guidance Rescale，范围 `0` ~ `1`。 |
| `noise_schedule` | string | 否 | Noise Schedule，例如 `karras`。 |
| `characters` | array | 否 | 多角色定义，见第 7 节。 |
| `use_coords` | boolean | 否 | 多角色坐标模式开关，见第 7 节。**只接受 `true` / `false`**，传字符串或其他类型会 400。 |
| `use_order` | boolean | 否 | 多角色按顺序排序开关，见第 7 节。 |

> 内层 `model` 字段会被忽略（见 4.1）。
> 输出图片格式 `image_format` 是内层 JSON 字段，与 `prompt` / `size` 平级，详见第 11 节。

完整绘图参数示例：

```json
{
  "prompt": "1girl, solo, masterpiece, best quality, detailed eyes",
  "negative_prompt": "lowres, bad anatomy, bad hands, text, watermark",
  "size": [832, 1216],
  "steps": 23,
  "scale": 5,
  "sampler": "k_euler_ancestral",
  "seed": 123456789,
  "variety_boost": true,
  "cfg_rescale": 0.5,
  "noise_schedule": "karras"
}
```

## 6. 尺寸

`size` 只能传整数数组 `[width, height]`：

```json
{ "size": [832, 1216] }
```

**字符串写法（包括 `"832x1216"`、`"portrait"` 等预设名）一律返回 400**，错误信息会提示使用 `[832, 1216]`。

| 用途 | 数组值 |
| --- | --- |
| 竖图 | `[832, 1216]` |
| 横图 | `[1216, 832]` |
| 方图 | `[1024, 1024]` |

当前最大限制：

| 类型 | 最大尺寸 |
| --- | --- |
| 竖图 | `[832, 1216]` |
| 横图 | `[1216, 832]` |
| 方图 | `[1024, 1024]` |

宽高必须是 64 的倍数，超过限制会返回 400。

## 7. 多角色（characters）

NovelAI V4 系列模型支持在同一张图中定义多个角色，每个角色有独立的正向 / 负向提示词和位置。

**适用模型**：建议使用 `nai-diffusion-4-*` 与 `nai-diffusion-4-5-*`。早期模型对该字段支持有限。

### 7.1 顶层开关

| 字段 | 类型 | 含义 |
| --- | --- | --- |
| `use_coords` | boolean | `true` = 使用 `position` 手动定位；`false`（默认）= 由模型自动安排。 |
| `use_order` | boolean | 是否把数组顺序作为提示词强度顺序。 |


### 7.2 `characters[i]` 字段

| 字段 | 类型 | 默认 | 说明 |
| --- | --- | --- | --- |
| `prompt` | string | 必填 | 该角色正向提示词（英文）。 |
| `negative_prompt` | string | `""` | 该角色负向提示词（英文）。 |
| `position` | string | `"C3"` | **5×5 网格预设字符串**，格式 `[A-E][1-5]`：字母列 A→E 由左到右，数字行 1→5 由上到下，中心 = `"C3"`。只接受字符串。 |

`position` 网格直观对照（横轴 = 字母 A→E 表示左右，纵轴 = 数字 1→5 表示上下）：

```
         A      B      C      D      E       ← 字母列（左 → 右）
      ┌──────┬──────┬──────┬──────┬──────┐
  1   │  A1  │  B1  │  C1  │  D1  │  E1  │   上
  2   │  A2  │  B2  │  C2  │  D2  │  E2  │
  3   │  A3  │  B3  │  C3  │  D3  │  E3  │   中
  4   │  A4  │  B4  │  C4  │  D4  │  E4  │
  5   │  A5  │  B5  │  C5  │  D5  │  E5  │   下
      └──────┴──────┴──────┴──────┴──────┘
   ↑
 数字行（上 → 下）
```

常用：左 = `B3`，右 = `D3`，中 = `C3`，左上 = `B2`，右下 = `D4`。

### 7.3 示例：左蓝右白双猫娘

下面这个 payload 用对角错位构图把两个角色分别钉在 `B2`（左上）和 `D4`（右下），并通过各自的 `negative_prompt` 互斥发色，方便肉眼判定多角色绑定与坐标定位是否生效。

```json
{
  "prompt": "2girls, cat girls, standing side by side, looking at viewer, masterpiece, best quality, detailed background, indoor, simple background",
  "negative_prompt": "lowres, bad anatomy, bad hands, text, watermark, blurry, extra fingers, deformed, jpeg artifacts, multiple views",
  "size": [1216, 832],
  "steps": 28,
  "scale": 5,
  "sampler": "k_euler_ancestral",
  "seed": 424242,
  "noise_schedule": "karras",
  "image_format": "png",
  "use_coords": true,
  "characters": [
    {
      "prompt": "1girl, cat girl, blue hair, long blue hair, blue cat ears, blue tail, blue eyes, blue ribbon, blue dress, holding a blue rose, on the left side",
      "negative_prompt": "white hair, pink hair, red hair",
      "position": "B2"
    },
    {
      "prompt": "1girl, cat girl, white hair, long white hair, white cat ears, white tail, red eyes, white kimono, holding a white fan, on the right side",
      "negative_prompt": "blue hair, pink hair, black hair",
      "position": "D4"
    }
  ]
}
```

**生效路径与判定**：

1. 顶层 `prompt` 描述**全局画面**（场景、构图、画质），不写任何角色专属特征。
2. 每个 `characters[i].prompt` 单独绑定**该角色专属特征**（发色、眼色、服装、手持物）。
3. `use_coords: true` + 每个角色的 `position` 把"哪个角色的特征落到画面的哪个区域"显式钉死；本例 `B2` 落在画面左上、`D4` 落在画面右下，故意拉开纵向距离来检验上下定位。
4. 各角色 `negative_prompt` 把**对方的颜色**列为禁止项，避免特征互相污染。

判定生效：左上区域应是蓝发蓝眼蓝裙拿蓝玫瑰，右下区域应是白发红眼白和服拿白扇。如果两边出现混色、同色双子、左右站位互换，或两个角色都挤在同一高度，就说明多角色绑定 / 坐标未生效。

## 10. max_tokens 预算

请求里的 `max_tokens` 或 `max_completion_tokens` 表示本次请求允许消耗的最大预算。

换算关系：

```text
1 Anlas = 10000 tokens
```

| 最大 Anlas | `max_tokens` |
| --- | --- |
| 1 | `10000` |
| 5 | `50000` |
| 10 | `100000` |

不传时按服务端默认上限 `100000`（10 anlas）。如果**预估消耗 > `max_tokens`**，会直接返回 400，不会真的调上游计费。

| --- | --- | --- | --- |
| `image_format` | string | `"png"` | 只接受 `"png"` 或 `"webp"`，其他值 400。 |

## 13. 解析图片

```python
import re
import json

_IMG_RE  = re.compile(r"!\[[^\]]*\]\((data:image/[^;)]+;base64,[^)]+)\)")
_SEED_RE = re.compile(r"<!--\s*seeds:(\[.*?\])\s*-->")

def extract_image_data_uris(content: str) -> list[str]:
    return _IMG_RE.findall(content)

def extract_seeds(content: str) -> list[int | None]:
    m = _SEED_RE.search(content)
    if not m:
        return []
    return json.loads(m.group(1))
```

把 data URI 落盘成 PNG / WebP：

```python
import base64
from pathlib import Path

def save_data_uri(data_uri: str, out_path: Path) -> None:
    # data:image/png;base64,xxxx... → 拆 header / payload
    header, _, payload = data_uri.partition(",")
    fmt = header.split(";", 1)[0].split("/", 1)[1]  # "png" / "webp"
    out_path.with_suffix(f".{fmt}").write_bytes(base64.b64decode(payload))
```

## 15. Python 示例

依赖：`pip install openai`（或 `uv add openai`）。SDK 版本 ≥ 1.0。

```python
import json
import re

from openai import OpenAI

OPENAI_BASE_URL = "服务提供方给你的 NewAPI 地址"
OPENAI_API_KEY  = "服务提供方给你的 NewAPI 密钥"

_IMG_RE  = re.compile(r"!\[[^\]]*\]\((data:image/[^;)]+;base64,[^)]+)\)")
_SEED_RE = re.compile(r"<!--\s*seeds:(\[.*?\])\s*-->")

client = OpenAI(base_url=f"{OPENAI_BASE_URL}/v1", api_key=OPENAI_API_KEY)


def generate_image() -> dict:
    draw_params = {
        "prompt": "1girl, solo, masterpiece, best quality, detailed eyes",
        "negative_prompt": "lowres, bad anatomy, bad hands, text, watermark",
        "size": [832, 1216],
        "steps": 23,
        "scale": 5,
        "sampler": "k_euler_ancestral",
        "image_format": "png",
    }
    # 注意：messages[0].content 必须是 json.dumps 后的字符串，不是 dict
    resp = client.chat.completions.create(
        model="nai-diffusion-4-5-full",
        messages=[
            {"role": "user", "content": json.dumps(draw_params)},
        ],
        stream=False,
        max_tokens=100000,
        timeout=180,
    )
    content = resp.choices[0].message.content or ""
    seed_match = _SEED_RE.search(content)
    return {
        "images": _IMG_RE.findall(content),
        "seeds": json.loads(seed_match.group(1)) if seed_match else [],
        "usage": resp.usage.model_dump() if resp.usage else None,
    }
```

错误处理：上游错误信封会被 `openai` SDK 包装成 `openai.APIStatusError`，其 `.status_code` 与 `.response.json()["error"]` 对应文档 §17 的 HTTP 状态码与错误信封。

## 16. 流式响应

如需流式响应：

```json
{ "stream": true }
```

**`stream` 只接受布尔**（`true` / `false` / 省略）。传字符串 `"true"` / `"false"`、数字
`1` / `0`、`"yes"` 等一律 400。

如果传 `stream: true`，服务端会按 OpenAI SSE 风格返回 `text/event-stream`。当前绘图渠道的

## 18. 前端表单建议

建议提供：

- 模型选择：从 `/v1/models` 获取。
- 正向 prompt：英文输入。
- 负向 prompt：英文输入。
- 尺寸：竖图 / 横图 / 方图（实际下发为 `[W, H]` 数组）。
- 张数：当前只允许 `1`，建议前端不提供"多张"选择。
- steps：`1` ~ `28`，默认 `23`。
- seed：可选。
- 采样器：默认 `k_euler_ancestral`。
- 多角色（可选）：每个角色独立 prompt / negative prompt / 位置；可切换"自动布局 / 手动坐标"。
- 最大预算：转换成 `max_tokens`（1 anlas = 10000 tokens）。

提交前建议校验：

- prompt 不为空。
- prompt、negative prompt 以及每个 `characters[i].prompt` / `characters[i].negative_prompt` 不包含 CJK 字符与全角符号。
- `steps` 不超过 `28`。
- `n_samples` 固定为 `1`，或不提供该字段。
- `size` 为整数数组 `[W, H]`，不超过服务限制。

## 19. 最小调用流程

1. 使用服务提供方给出的 NewAPI 地址和 NewAPI 密钥。
2. 调用 `/v1/models` 获取可用模型，选一个赋给外层 `model`。
3. 组装绘图参数对象 `drawParams`（**不写 `model`**，由外层提供），需要特定输出格式时在里面加 `image_format: "png" | "webp"`。
4. `json.dumps(draw_params)` 后放入 `messages[0].content`。
5. `POST /v1/chat/completions`。
6. 从 `choices[0].message.content` 提取 markdown 中的 data URI。
7. 显示或下载图片，必要时从注释里读 `seeds`。

## 20. 图生图（i2i / inpaint / vibe transfer / character reference）

以下 4 个字段都是内层 payload 字段，与 `prompt` / `size` 平级。它们让生成过程接受**额外的参考图**作为输入。所有图片字段都接受 base64 字符串或 `data:image/<png|webp>;base64,...` 形式的 data URI。

互斥规则（同时违反会直接 400）：

- `i2i` 与 `inpaint` **互斥**：同一请求只能选其一。
- `controlnet`（Vibe Transfer）与 `character_references` **互斥**：同一请求只能选其一。
- 其他组合都允许，例如 `i2i + controlnet`、`inpaint + character_references`。

### 20.1 `i2i`（图生图）

基于一张参考图重绘整张图，由 `strength` 控制保留多少原图结构。

| 字段 | 类型 | 范围 / 默认 | 说明 |
| --- | --- | --- | --- |
| `i2i.image` | string（base64 / data URI） | 必填 | 源图。 |
| `i2i.strength` | number | `0.01` ~ `0.99`，默认 `0.7` | 变换强度，越小越像原图。 |
| `i2i.noise` | number | `0.0` ~ `0.99`，默认 `0.0` | 注入噪声量。 |
| `i2i.seed` | number | 默认随机 | i2i 噪声种子。 |

图片约束：

- 必须能被解码为合法图片。
- 宽高必须是 64 的倍数、不超过平台尺寸上限。
- **图片宽高必须与外层 `size` 严格相等**，不一致直接 400 。

示例：

```json
{
  "prompt": "1girl, white dress",
  "size": [832, 1216],
  "i2i": {
    "image": "data:image/png;base64,iVBORw0KGgo...",
    "strength": 0.5,
    "noise": 0
  }
}
```

### 20.2 `inpaint`（局部重绘）

基于 mask 仅重绘指定区域。模型名仍传基础模型（如 `nai-diffusion-4-5-full`），服务端会自动切换到对应的 inpainting 变体。

| 字段 | 类型 | 范围 / 默认 | 说明 |
| --- | --- | --- | --- |
| `inpaint.image` | string（base64 / data URI） | 必填 | 原图。 |
| `inpaint.mask` | string（base64 / data URI） | 必填 | 遮罩图：**白色 = 需要重绘的区域，黑色 = 保留原图**。 |
| `inpaint.strength` | number | `0.01` ~ `1.0`，默认 `1.0` | 重绘强度。 |
| `inpaint.seed` | number | 默认随机 | 噪声种子。 |

图片约束：

- `image` 与 `mask` 都必须可解码、宽高 64 倍数、不超过平台尺寸上限。
- `image`、`mask`、外层 `size` **三者宽高必须完全相等**，任一不一致都会 400。
- 不支持 `noise` 字段（与 `i2i` 的差异）。

示例：

```json
{
  "prompt": "1girl, red dress",
  "size": [832, 1216],
  "inpaint": {
    "image": "data:image/png;base64,iVBORw0KGgo...",
    "mask":  "data:image/png;base64,iVBORw0KGgo...",
    "strength": 1.0
  }
}
```

### 20.3 `controlnet`（Vibe Transfer，风格 / 氛围参考）

把一到多张参考图的整体风格 / 氛围迁移到生成图上。每张图独立配置 `info_extracted`（提取信息强度）和 `strength`（单图参考强度）。

| 字段 | 类型 | 范围 / 默认 | 说明 |
| --- | --- | --- | --- |
| `controlnet.images` | array | 必填，**最多 4 张** | 参考图列表。 |
| `controlnet.images[i].image` | string（base64 / data URI） | 必填 | 单张参考图。 |
| `controlnet.images[i].info_extracted` | number | `0.01` ~ `1.0`，默认 `0.7` | 从该图提取的信息量。 |
| `controlnet.images[i].strength` | number | `0.01` ~ `1.0`，默认 `0.6` | 该图单独的参考强度。 |
| `controlnet.strength` | number | `0.0` ~ `1.0`，默认 `1.0` | 整体 ControlNet 强度叠加系数。 |

图片约束：

- 仅校验能被解码为合法图片。
- **不限制边长**：NovelAI 服务端会自行 resize / pad 参考图，调用方可以传任意尺寸。
- 张数 `> 4` 直接 400。
- 与 `character_references` 互斥。

示例（两张参考图）：

```json
{
  "prompt": "1girl, city street, night",
  "size": [832, 1216],
  "controlnet": {
    "strength": 1.0,
    "images": [
      { "image": "data:image/png;base64,...", "info_extracted": 0.7, "strength": 0.6 },
      { "image": "data:image/png;base64,...", "info_extracted": 0.5, "strength": 0.4 }
    ]
  }
}
```

#### 20.3.1 `cache_id` 复用：免传图、免编码计费

NovelAI 对 vibe 编码是按次计费的。本网关把每次成功编码出的 vibe data 落库，并对外暴露一个 `cache_id`（22 字符 URL-safe base64）。下次同一张图 / 同 `info_extracted` / 同 `model` 的请求**可改用 `cache_id` 代替图片字节**，跳过上传与编码计费。

**响应回传 cache_id**：网关在 `choices[0].message.content` 末尾以 HTML 注释形式追加 `vibe_cache_ids` 注释，`index` 与本次请求的 `controlnet.images[i]` 一一对应。完整响应形态示例（节选关键字段）：

```json
{
  "id": "bestnai-3a4c...",
  "object": "chat.completion",
  "created": 1710000000,
  "model": "nai-diffusion-4-5-full",
  "choices": [
    {
      "index": 0,
      "message": {
        "role": "assistant",
        "content": "![image_0](data:image/png;base64,iVBORw0KGgo...)\n<!-- seeds:[123456789] -->\n<!-- vibe_cache_ids:[{\"index\":0,\"cache_id\":\"AbCdEfGhIjKlMnOpQrStUv\"},{\"index\":1,\"cache_id\":\"ZyXwVuTsRqPoNmLkJiHgFe\"}] -->"
      },
      "finish_reason": "stop"
    }
  ],
  "usage": {
    "prompt_tokens": 1,
    "completion_tokens": 30000,
    "total_tokens": 30001
  }
}
```

也就是说，`message.content` 实际内容里以换行分隔依次出现：

```text
![image_0](data:image/png;base64,...)
<!-- seeds:[123456789] -->
<!-- vibe_cache_ids:[{"index":0,"cache_id":"AbCdEf..."},{"index":1,"cache_id":"ZyXwVu..."}] -->
```

`vibe_cache_ids` 注释里 `cache_id` 是 22 字符 URL-safe base64 字符串。客户端应自行解析并存储，下次请求复用。

**Python 解析示例**：

```python
import json
import re

_VIBE_RE = re.compile(r"<!--\s*vibe_cache_ids:(\[.*?\])\s*-->")

def extract_vibe_cache_ids(content: str) -> list[dict]:
    """从 chat completion 的 message.content 中解析 vibe_cache_ids。

    返回形如 [{"index": 0, "cache_id": "AbCdEf..."}, ...]；
    无该注释时返回空列表。
    """
    m = _VIBE_RE.search(content)
    if not m:
        return []
    return json.loads(m.group(1))


# 使用：拿到 cache_id 后按 index 映射回本地 controlnet.images
content = resp["choices"][0]["message"]["content"]
for entry in extract_vibe_cache_ids(content):
    print(entry["index"], entry["cache_id"])
```

**下次请求改用 cache_id**：把 `controlnet.images[i]` 从原本的 `{image, info_extracted, strength}` 改写为 `{cache_id, strength?}`：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `controlnet.images[i].cache_id` | string | 必填 | 上次响应里返回的 `cache_id`。 |
| `controlnet.images[i].strength` | number | 可选 | 该图单独的参考强度，规则同上。 |

**严格二态**：单条 `controlnet.images[i]` **要么是 `image+info_extracted` 完整态，要么是 `cache_id` 复用态，不允许混合**。混用任意额外字段（含 `image`、`info_extracted`）一律 400。`cache_id` 必须为非空字符串。

**跨模型保护**：网关会校验 `cache_id` 对应的缓存 `model` 与请求外层 `model` 一致，不一致直接 400，防止跨模型用同一份 vibe data。

**未命中**：`cache_id` 在服务端找不到记录 → 400（不会静默回退到编码模式，避免静默扣费）。

#### 20.3.2 Vibe Transfer 流量附加费（surcharge）

只要本次请求的 `controlnet.images` **存在至少一张通过图片字节传入的参考图**（即非全量 `cache_id` 复用），网关就额外收取 **1 anlas** 流量附加费，用以覆盖图片传输与编码成本。

- 全量 `cache_id` 复用：**不**收附加费。
- 任意一张走完整态：收 1 anlas（按请求计，不按图数）。
- 该附加费会体现在 `usage.completion_tokens`（含在网关真实扣减里），客户端可据此对账。

### 20.4 `character_references`（角色参考 / Precise Reference）

V4.5 系列模型的角色参考能力，把指定参考图的角色形象 / 风格作为强约束注入生成。

| 字段 | 类型 | 取值 / 默认 | 说明 |
| --- | --- | --- | --- |
| `character_references` | array | **最多 1 张** | 角色参考图列表。 |
| `character_references[i].image` | string（base64 / data URI） | 必填 | 参考图。 |
| `character_references[i].type` | string | `"character"` / `"style"` / `"character&style"`，默认 `"character&style"` | 提取目标：仅角色、仅风格、或同时。 |
| `character_references[i].fidelity` | number | `0.0` ~ `1.0`，默认 `1.0` | 保真度（次要强度）。 |
| `character_references[i].strength` | number | `0.0` ~ `1.0`，默认 `1.0` | 主参考强度。 |

图片约束：

- 仅校验能被解码为合法图片。
- **不限制边长**：NovelAI 服务端会自动把参考图 upscale / downscale + pad 到 `1024x1536` / `1472x1472` / `1536x1024` 三种规范尺寸之一。
- 张数 `> 1` 直接 400。
- 与 `controlnet` 互斥。

示例：

```json
{
  "prompt": "1girl, standing in a forest",
  "size": [832, 1216],
  "character_references": [
    {
      "image": "data:image/png;base64,...",
      "type": "character",
      "fidelity": 1.0,
      "strength": 1.0
    }
  ]
}
```
